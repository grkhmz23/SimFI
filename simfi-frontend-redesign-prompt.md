# SimFi Frontend Redesign — Kimi Code Task

## Who you are and what you're doing

You are rebuilding the entire frontend of **SimFi**, a multi-chain (Base + Solana) paper trading platform for memecoins. The backend is production-ready and must not be touched. The current frontend is AI-generated-looking, generic, and unprofessional. Your job is to replace it with an **editorial-luxury, investor-grade interface** without breaking a single API contract.

This is a pre-investor sprint. The next viewer of this codebase is a Base ecosystem investor. The UI must look like it was designed by a studio, not generated by a prompt.

---

## Non-negotiables (read first, do not violate)

1. **No placeholders, no TODOs, no stubs, no pseudo-code, no "example only" code in production paths.** Every file you write must compile and run. Empty functions, `return null`, `// implement later` — all forbidden.
2. **Do not invent backend endpoints, payloads, or response shapes.** If you are unsure what an endpoint returns, read the backend source (`server/routes.ts`, `server/services/*.ts`, `shared/schema.ts`) and match it exactly.
3. **Do not touch the backend** (`server/`, `shared/`, `drizzle.config.ts`, migrations) except to read it. If something is missing on the server, stop and flag it. Do not add server code.
4. **Preserve auth flow:** JWT HttpOnly cookies with Authorization header fallback. Keep existing login/register/logout semantics.
5. **Preserve server-authoritative pricing:** the client never decides execution price. Keep idempotency key headers on trades (`X-Idempotency-Key`).
6. **Preserve existing routes** (wouter paths). URLs users have bookmarked must keep working.
7. **Environment variables only for config.** No hardcoded secrets, no fake values that pretend to work. Update `.env.example` if you add anything.
8. **Complete files only.** When you output code, output the entire file, not a fragment.

## Phase 2 — Design system (before rebuilding any page)

Produce `DESIGN_SYSTEM.md` and a tokens file. This is the spec you will build against.

### Aesthetic direction (lock this in)

**Editorial luxury, not Web3 neon.** Reference points to internalize:

- **Linear** — restrained, mono numbers, dark, every pixel considered.
- **Stripe** — editorial typography, generous whitespace.
- **Arc browser** — tasteful motion, depth via translucency done right.
- **Financial Times digital** — editorial data presentation, tabular figures.
- **Matches Fashion / Ssense** — dark luxury, serif display, product-as-hero.
- **Bloomberg Terminal reimagined for 2026** — data density with breathing room.

**Explicitly forbidden references:** pump.fun, generic shadcn templates, purple-blue Web3 gradients, cyberpunk, synthwave, Discord aesthetic, neon green/red P&L, emoji-as-iconography, stock illustration packs, generic hero gradients.

### Color tokens

Build around a **warm near-black** base, not pure black. Approximate targets (tune in the tokens file):

- `--bg-base`: warm near-black, ~`#0a0a0b`
- `--bg-raised`: graphite, ~`#141416`
- `--bg-overlay`: layered translucent
- `--border-subtle`: ~`rgba(255,255,255,0.06)`
- `--border-strong`: ~`rgba(255,255,255,0.12)`
- `--text-primary`: warm ivory, ~`#f5f3ee`
- `--text-secondary`: ~`#9a9894`
- `--text-tertiary`: ~`#5f5d58`
- `--accent-gain`: muted emerald, ~`#3fa876` (NOT neon green)
- `--accent-loss`: oxblood/crimson, ~`#c24d4d` (NOT neon red)
- `--accent-premium`: champagne, ~`#c9a96e` (use sparingly — badges, top leaderboard, premium moments only)
- No gradients except extremely subtle (≤8% contrast shift).

### Typography

Three-family system, loaded via `@fontsource` or Google Fonts (pick one stack, be consistent):

- **Display (serif, editorial):** `Fraunces`, `Tiempos`, or `Instrument Serif` — used for page titles and hero moments only.
- **UI (sans):** `Inter` with `font-feature-settings: 'ss01', 'cv11'` enabled, or `Söhne` if licensed. Tabular figures everywhere numbers appear: `font-variant-numeric: tabular-nums`.
- **Mono (numbers, addresses, timestamps):** `JetBrains Mono` or `IBM Plex Mono`. Tabular by default.

Type scale: editorial, not utilitarian. Display sizes are large (clamp(2.5rem, 6vw, 5rem) for hero). Body 15–16px. Numbers always mono.

### Spacing / grid

- 8-point spacing scale.
- Max content width: 1280px. Hero sections can breathe wider.
- Generous vertical rhythm — sections breathe.
- Data-dense tables are allowed, but never feel cramped.

### Motion principles

- **Framer Motion**, tastefully. No bouncy springs on data. Use `ease: [0.16, 1, 0.3, 1]` (custom cubic-bezier, feels expensive).
- Hover states on interactive elements only. No hover-everywhere.
- Page transitions: subtle fade + 4px translate, 240ms.
- Number counters on P&L and balance: `framer-motion` `useMotionValue` + `animate` — *tasteful*, not casino-style.
- **GSAP** is permitted for the landing hero only if a timeline is needed (see your Maniar Atelier work for reference tone).

### Surfaces (glassmorphism done right)

- Use backdrop-filter sparingly, only on floating overlays (TradeModal, dropdowns, sticky navs on scroll).
- Not every card is glass. Most cards are solid `--bg-raised` with a subtle 1px `--border-subtle`.

### Iconography

- **Lucide React only.** Stroke width 1.5. Size 16 or 20 by default. Never mix icon libraries.
- No emoji in UI chrome. Emoji in user-generated content is fine.

### Imagery

- Token logos come from DexScreener response — render them at a consistent size with a subtle rounded mask and fallback initial.
- No stock illustrations. No generic "crypto art".
- Landing hero may use a WebGL shader (keep it restrained — think a single slow-moving gradient mesh, not a particle system).

### Deliverables for Phase 2

- `DESIGN_SYSTEM.md` with every decision above locked in, including exact hex values and font choices.
- `client/src/styles/tokens.css` — CSS custom properties for all tokens.
- `tailwind.config.ts` extended to read from tokens (don't hardcode colors in Tailwind config; reference CSS variables).
- `client/src/styles/typography.css` — font imports, feature settings, type scale utilities.

Commit these before Phase 3.

## Phase 4 — Page rebuilds (in this order)

Rebuild pages in this order. After each page, verify it against the backend contract (run the app, hit real endpoints, confirm data flows). Do not batch them.

### 4.1 Shared layout + navigation

- Top nav: wordmark left, primary routes center, chain switcher + user menu right.
- Chain switcher: segmented control, not a dropdown. Base / Solana.
- Search: cmd+K global search (DexScreener search on the current chain). This is a hero feature — build it well. Results dropdown with token logo, symbol, name, chain badge, market cap, 24h change.
- Mobile: bottom tab bar, not hamburger.
- Footer: minimal — legal links, status indicator (`/api/health`).

### 4.2 Landing page (`/`)

This is the investor's first impression.

- Hero: editorial display type, short sharp value prop, one primary CTA ("Start Trading"), one secondary ("How it Works").
- Optional WebGL background: a single slow gradient mesh shader (reuse your Swarp aether shader tuning if it fits). Must be `prefers-reduced-motion` aware — disable animation if set.
- Below fold: three sections — "Trade without risk", "Learn the trenches", "Compete and climb". Each with a real screenshot/mock of the relevant feature, not an illustration.
- Live leaderboard teaser: show current 6h period top 3 from `/api/leaderboard/current?chain=base` (verify exact path in audit).
- Stats strip: total trades executed, active traders, tokens tracked — pull from real endpoints if they exist; if not, flag it in audit and do not fabricate numbers.

### 4.3 Trade page (`/trade`)

- Two-pane layout on desktop: token list left (trending, new pairs, hot, filtered by chain), chart + trade panel right.
- Token list rows: logo, symbol + name, market cap, 24h change (mono, tabular, color token), liquidity indicator.
- Chart: `lightweight-charts`, dark theme matching tokens, minimal gridlines.
- Trade panel: amount input, buy/sell toggle, balance readout (mono), estimated output, slippage note, submit.
- Below the submit button, leave a **clearly marked slot** (an empty container with a comment: `{/* RISK_SCORE_CARD_SLOT — future educational feature */}`) so the risk-scoring card can drop in later without a refactor.

### 4.4 Token detail page (`/token/:address`)

- Hero strip: token logo, symbol, name, chain chip, contract address pill, DexScreener/explorer links.
- Data row: price (large mono), 24h change, market cap, liquidity, volume — all `DataCell`s.
- Chart: `lightweight-charts`, full-width.
- Trade CTA: opens TradeModal.
- Tabs below: Overview, Holders (if endpoint exists), Recent Trades (if endpoint exists). Only build tabs for endpoints that actually exist — verify in audit.

### 4.5 Portfolio (`/portfolio`)

- Summary cards at top: total value, total invested, total P&L (realized + unrealized), win rate. All mono numbers.
- Balance-over-time chart (recharts, single line, minimal).
- Open positions table: token, entry, current, qty, value, unrealized P&L. Sortable columns.
- Pagination or infinite scroll — match what the backend supports (verify in audit).

### 4.6 Positions (`/positions`) and History (`/history`)

- Dedicated pages, not just filtered portfolio views.
- History: closed trades with P&L, hold time, chain badge.

### 4.7 Leaderboard (`/leaderboard`)

- Three tabs: Current period (6h live), All-time, Past winners.
- Top 3 get premium accent (champagne). Rest are standard.
- Row: rank, username, realized profit, trade count, chain.
- Countdown timer to next period close (mono, prominent).

### 4.8 Trader profile (`/trader/:username`)

- Public profile: username, join date, rank badge, follow button.
- Stats: total P&L, win rate, avg hold time, best trade, worst trade.
- Recent trades feed.
- Achievement badges earned (grid).

### 4.9 Auth pages (`/login`, `/register`)

- Split layout: form left, brand panel right (editorial, not a stock image).
- Register: include referral code field.
- Error states: inline, not toasts.

### 4.10 Settings (`/settings`)

- Profile, referral code, notifications, data export if endpoint exists. Keep minimal.
